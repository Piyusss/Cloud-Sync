# Project: Cloud Sync (Online Cloud Storage Service)

## Goal

Build a cloud storage platform where users can:

* Upload files
* Download files
* Organize folders
* Share files
* Resume interrupted uploads
* Version files
* Deduplicate storage
* Handle large files efficiently

**Tech Stack**

Frontend:

* React
* TypeScript
* Tailwind
* React Query

Backend:

* Java 21
* Spring Boot 3
* Spring Security
* JWT

Database:

* PostgreSQL

Storage:

* MinIO (S3 compatible)

Cache:

* Redis

Deployment:

* Docker
* VPS or cloud VM

# Phase 4 — Chunk Uploads (Week 3)

This is where it becomes interesting.

## Why

Uploading:

```text
2 GB movie
```

should not fail at 99%.

---

### Upload Flow

Frontend:

```text
Split file into chunks
```

Example:

```text
2 GB

Chunk 1
Chunk 2
Chunk 3
...
Chunk 200
```

Upload chunks independently.

Backend:

```http
POST /chunks
POST /chunks/complete
```

Store temporary chunks.

Merge after completion.

---

### Benefits

* Resume uploads
* Retry failed chunks
* Parallel uploads

This is a major interview talking point.

# Phase 6 — Storage Deduplication (Week 5)

Very impressive feature.

### Problem

100 users upload:

```text
ubuntu.iso
```

Store only once.

---

### Solution

Generate SHA-256 hash.

```text
Hash
=
A1B2C3...
```

Before upload:

Check:

```sql
SELECT *
FROM physical_files
WHERE hash = ?
```

If exists:

```text
Create metadata only
```

No actual upload.

---

Tables

```sql
physical_files
--------------
id
hash
size
storage_key
ref_count
```

# Phase 15 — Differential Sync (Expert Level)

Like Dropbox.

Only changed blocks are uploaded.

Example:

```text
500 MB file
```

Change:

```text
1 MB
```

Upload:

```text
1 MB
```

instead of:

```text
500 MB
```

This is a genuinely difficult engineering problem.

# Resume Impact

Most candidates:

```text
Built Google Drive Clone
```
Looks ordinary.

Instead write:
> Designed and developed a cloud storage platform supporting chunked uploads, file versioning, SHA-256-based deduplication, resumable transfers, secure share links, Redis caching, and distributed object storage using MinIO and Spring Boot.
That description alone is likely to trigger deeper technical questions in interviews.
One more suggestion: maintain a design document in your repo explaining upload flow, deduplication flow, database schema, and scaling strategy. Many candidates build projects; far fewer demonstrate engineering design thinking.
